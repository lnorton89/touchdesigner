<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Demo user input to Router | Prompt DAT edits and parameter pulses initiate outbound model requests. |
| Callback payload to downstream TD logic | Result data is exposed for other operators to react to. |
| Demo docs to future distributable component | Example instructions can accidentally preserve local machine assumptions. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-09 | Information Disclosure | Demo README and callback script | high | mitigate | Use no API key values, no private paths, and no saved machine-specific configuration; document local setup assumptions for Phase 4. |
| T-01-10 | Elevation of Privilege | Demo callback target | medium | mitigate | Demo callback accepts structured payload only and does not evaluate text from the provider or prompt. |
| T-01-11 | Denial of Service | Slow/error demo cases | medium | mitigate | Keep slow proof bounded with finite timeout or simulated delay and confirm frame counter continues during running state. |
| T-01-12 | Spoofing | Status and callback payloads | low | mitigate | Include request_id, trigger_source, provider, model, and status in visible payloads so downstream operators can distinguish request origins. |
</threat_model>
