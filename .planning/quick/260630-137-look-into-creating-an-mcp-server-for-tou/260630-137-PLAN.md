<objective>
Investigate feasibility and architecture for creating an MCP (Model Context Protocol) server
that exposes TouchDesigner 2022 capabilities as MCP tools/resources. Produce a research document
with concrete architecture recommendations and a minimal working server sketch.

Purpose: Before committing engineering effort, validate that the MCP Python SDK (`mcp` package,
Python 3.10+) can coexist with TD 2022's Python 3.11 environment, that a transport choice exists
which does not freeze the TD main thread, and that the concept of exposing TD operators as MCP
tools is architecturally sound.

  mapping, dependency implications for the existing external-venv strategy, concrete recommendations

  Python 3.11) that exposes 2-3 dummy tools mimicking TD capabilities (e.g., `td_get_parameter`,
  `td_get_dat_text`, `td_pulse_trigger`) with realistic input/output shapes — verifiable via
  `curl` / MCP Inspector

    1. **SDK Compatibility** — Confirm that the `mcp` PyPI package (modelcontextprotocol/python-sdk v1.x)
       supports Python 3.11 (TD 2022's shipped Python). Check its dependency tree (pydantic, httpx, anyio/uvicorn)
       for conflicts with TD's constrained environment. Note the `requires-python = ">=3.10"` constraint.

    2. **Transport Analysis** — Evaluate three hosting patterns:
       - **Embedded thread + streamable-http**: MCP server runs as a daemon thread inside TD's Python process,
         serving via uvicorn on localhost. Risk: can TD's Python host an ASGI server without frame interference?
       - **Companion process**: MCP server runs as a standalone Python script in the external venv, communicates
         with TD via a local socket/pipe/HTTP API. Risk: two-process coordination complexity.
       - **stdio transport** (rejected): TD owns process stdin/stdout — stdio-based MCP is incompatible with
         TD's execution model. Document why this approach fails.

    3. **TD Capability Mapping** — Map TD operator concepts to MCP primitives:
       - **Tools** (callable functions): get/set parameter, read DAT text, read CHOP channel, trigger pulse,
         list network children, cook operator.
       - **Resources** (readable data): parameter snapshot, DAT content, CHOP data as JSON, node graph
         topology, current frame/time.
       - **Prompts** (LLM prompt templates): "summarize this network", "debug this operator", "suggest
         connections" — worth exposing eventually but secondary to tools/resources.

    4. **Dependency Strategy** — How the `mcp` package fits into the existing external-venv approach.
       Does it require `uvicorn` (for streamable-http)? Any extras needed (`mcp[cli]`)?
       Cross-reference with the project's existing `httpx` and `pydantic` dependencies — are there version
       constraint collisions?

    5. **Recommendation** — Concrete recommendation: which hosting pattern to pursue, which transport,
       what the minimum viable MCP server for TD looks like, and what Phase-level work would be needed
       to integrate it with the existing ModelRouter/Agent components.

    Format RESEARCH.md with clear sections matching the above structure. Use tables for tradeoff comparisons.
